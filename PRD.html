1. What we are building

Aide is a fully voice-controlled web app. A blind user talks to an agent — Aide — that finds jobs suited to them, applies on their behalf, runs a spoken skills assessment, and receives and manages their pay through the Monnify API. The user speaks; Aide listens (speech-to-text), reasons, acts, and speaks back (text-to-speech). Nothing in the user's journey requires sight or a keyboard.

2. The problem

A blind Nigerian who can transcribe, translate, code, or do phone support is locked out of digital income — not by ability, but by interface. Job boards, application forms, skills tests, and the payment dashboards where money lands all assume you can see a screen and use a keyboard. The capability exists; the access does not.

The one-sentence pitch: "A blind graduate who transcribes faster than you can't finish a single job application — because every form, test, and payment screen assumes she can see it."
The enemy: the screen-and-keyboard assumption baked into every fintech and job platform.

4. Product decisions (locked)

4.1 "Voice-native" means the voice path is complete, not that no screen exists

The blind user can do everything by voice — sight is never required. A visual layer still exists, but it is not the primary user's dependency. It serves (a) low-vision users, with bold high-contrast type, and (b) the demo: an observer screen that shows the live transcript, current state, balance, and payment events so judges can see the loop working, not just hear it.

4.2 Delivery: web app + microphone (not telephony)

We build a browser app with a mic button. Telephony (call a number, no app) is the stronger long-term product and a great roadmap line, but it is the wrong build for an 8-day window: it is mostly plumbing, adds on-stage failure points, and — critically — has no screen for judges to follow. Web + mic is fastest to build, rock-solid to demo, and the screen doubles as the observer view. We pitch telephony as "what's next."

5. Scope

Must-have — this is the demo, build first

  1. Voice I/O loop — speech-to-text in, Aide reasons, text-to-speech out. < ~2s per turn.
  2. Aide agent with a fixed toolset (§7). Conversational on navigation, deterministic on money.
  3. Job discovery + application — a seeded set of accessible jobs; Aide reads them, the user picks, Aide applies.
  4. Oral assessment — Aide administers a short spoken skills test on the employer's behalf, scores it, marks the skill verified.
  5. Payment receipt (PROVEN) — employer funds land in the user's Monnify reserved account; Aide announces it only after server-side verification.
  6. Withdrawal — user says "send my money to my bank"; Aide does a name-enquiry read-back, then calls Monnify's disbursement (see §9 for demo vs production behaviour).

Nice-to-have — only if the loop above is solid

  • Application tracking by voice · balance/history queries · employer web portal to post jobs and confirm completion · platform fee via transaction splitting.

Cut without hesitation

  • Real accounts/KYC at scale · native mobile app · recurring/bulk transfers · multi-language (English + optional Pidgin only) · anything needing live Monnify keys.

6. Core user flows

F1 — Find work. "Aide, find me transcription jobs." → Aide lists 2–3 seeded jobs with pay + task. → "Apply to the second one." → Aide confirms and applies.

F2 — Prove the skill. "This job needs a transcription test. I'll play 20 seconds of audio; repeat what you hear." → captures the response → scores it → marks the skill verified.

F3 — Get paid. Employer marks the task complete and funds the reserved account (in the demo, via the Monnify web simulator). → Monnify confirms the payment → Aide: "You've been paid ₦12,000 for the transcription job."

F4 — Withdraw. "Send it to my bank." → Aide validates the account (name enquiry) and reads the name back for spoken confirmation → on "yes," calls the disbursement API → confirms status.

7. The Aide agent

Aide is an LLM with strict tool-calling. It never invents facts about money, jobs, or balances — it calls a typed server tool and speaks the result. The tools are the only actions it can take.

ToolDoesBacked by
listJobs(filter)Return seeded jobs matching a spoken filterDB
applyToJob(jobId)Create an application for the userDB
startAssessment(jobId)Begin the oral skills testDB + audio
submitAssessment(answer)Score the spoken answer, set skill verifiedDB + scoring
getBalance()Return verified balance (sum of PAID inbound)Monnify
getApplicationStatus()List the user's applications and statesDB
registerPayoutAccount(acct, bank)Name-enquiry + save the user's bank for read-backMonnify
initiateWithdrawal(amount)Call Monnify single-transferMonnify
confirmPayment()Poll/verify a landed payment before announcingMonnify
Guardrails (these are also scoring points — the challenge says "AI slop is frowned upon"):
  • Aide reports a payment, balance, or transfer only from a verified server call — never from its own reasoning or a raw webhook. The LLM narrates; it never decides financial truth.
  • Every money-moving action needs an explicit spoken confirmation, and Aide reads back the amount + destination name before executing.
  • If speech-to-text is unclear on a money command, Aide re-confirms rather than guessing.

8. Payment architecture (Monnify)

The platform is the Monnify merchant. Each worker gets a dedicated virtual account (a reserved NUBAN) that acts as their earnings account. Employers pay into it; Monnify notifies us; we verify and let Aide announce.

Verified endpoints (already wrapped in src/monnify.ts)

POST /api/v1/auth/login → bearer token (~1h, cached) POST /api/v2/bank-transfer/reserved-accounts → create earnings account GET /api/v1/bank-transfer/reserved-accounts/transactions → confirm inbound (PAID) GET /api/v2/transactions/{ref} → re-verify a transaction GET /api/v1/disbursements/account/validate → name enquiry (read-back) POST /api/v2/disbursements/single → withdrawal (needs activation) POST /api/v2/disbursements/single/validate-otp → authorize payout (server-side)

Sandbox mechanics: inbound "employer payments" are simulated via https://websim.sdk.monnify.com/?#/bankingapp — send a transfer to the reserved account (bank must match), which produces a real transaction we confirm through the API. Base URL https://sandbox.monnify.com.

9. Withdrawal design — and why the user never touches an OTP

Outbound payouts are 2FA-protected and, on our sandbox, gated behind full business activation (KYC + Monnify review). We handle this with an architecture that is both correct for production and demoable now: the platform (the merchant) authorizes the payout; the blind user only consents by voice. A user who cannot read an OTP off a screen never has to.

Demo now (no KYC)After business activation
Transfer callReal API call → returns PENDING_AUTHORIZATIONReal API call → platform authorizes OTP server-side
Aide says"Your withdrawal is on its way — I'll confirm when it lands.""Done. ₦12,000 sent to your GTBank account."
User handles an OTP?NoNo — the platform does

The security a blind user gets instead of a visual OTP: (1) voice consent, (2) Aide reads back the destination account name from name-enquiry so they hear exactly where the money is going, (3) the platform authorizes the settlement. "Security without sight."

Balance = sum of credit ledger entries with status PAID. Every entry is written only after Monnify confirms — the ledger is the single source of financial truth Aide reads from.

13. Accessibility principles (non-negotiable)

  • No task requires sight or a keyboard to complete, end to end. Test the demo screen-off.
  • Aide confirms before every irreversible/money action and reads back critical values.
  • Every failure has a spoken, human explanation — never a silent dead-end.
  • The visual layer is bold, high-contrast, large-type, and screen-reader friendly (for low-vision users).
